## 🏗️ System Architecture & State Management

The backend of the application is built on a sophisticated state machine using **LangGraph** and a robust **FastAPI** application. It orchestrates a multi-agent workflow that intelligently handles dynamic routing, web research, parallel writing, and automated image generation.

### Global State Management
The entire system operates on a central shared `State` dictionary. As execution moves through the system's graph, each node dynamically updates this state. This enables the workflow to seamlessly track crucial context throughout the process, including:
* The topic and routing mode.
* Generated search queries and collected evidence.
* The Orchestrator's structural plan.
* Dynamically generated markdown sections and image specifications (`ImageSpec`).

---

## 🤖 The Primary AI Graph (Node Workflow)

The execution flow is structured as a Directed Acyclic Graph (DAG) consisting of several specialized, autonomous nodes:

### 1. Router Node (Decision Engine)
The Router Node analyzes the user's topic and the requested "as-of" date to evaluate the temporal needs of the request. It categorizes the prompt into one of three distinct routing modes:
* **`closed_book`**: Used for stable, evergreen topics that require no external research or recent data.
* **`hybrid`**: Chosen for evergreen concepts that rely on stable foundational principles but still require up-to-date, real-world examples to supplement the draft.
* **`open_book`**: Triggered for highly volatile, news-based topics that demand immediate, highly recent data strictly gathered from the last 7 days.

*If the node determines that external research is necessary (i.e., hybrid or open_book), it generates 3 to 10 highly specific search queries to guide the data gathering phase.*

### 2. Research Node (Evidence Synthesis)
Triggered only when the Router requests external data, this node executes the generated search queries using the **Tavily Search API**. Rather than dumping raw search results directly into the workflow, the system processes the findings through an LLM to extract high-signal `EvidenceItem` objects containing:
* The source's title and URL.
* The publication date.
* A relevant text snippet.

*To ensure accuracy, the node filters out any outdated information by comparing the extracted publication dates against the strict recency limits established by the Router.*

### 3. Orchestrator Node (Senior Technical Editor)
Acting as the senior technical editor, the Orchestrator evaluates the user's topic alongside the gathered `EvidenceItem` objects to outline the blog post. It generates a strict **Plan** divided into 5 to 9 discrete tasks. Each task acts as an instruction manual for the drafting phase, detailing:
* A specific goal and key bullet points to cover.
* Strict target word counts.
* Boolean flags that explicitly dictate whether the assigned section requires code snippets or evidence citations.

### 4. Worker Nodes (Parallel Execution)
To handle the actual writing, the system utilizes the LangGraph `Send` API to simultaneously spin up a separate worker instance for each individual task generated by the Orchestrator. These parallel nodes:
* Strictly adhere to the specific target word counts assigned to their task.
* Enforce citation requirements by weaving the verified evidence URLs directly into the generated text as markdown citations.
* Include code snippets if dictated by the Orchestrator's boolean flags.

*Once complete, each worker returns its finalized markdown section back to the global shared state.*

## 🎨 Reducer Subgraph (Draft Finalization)

Once all parallel worker nodes have finished drafting, the graph funnels their output into the Reducer Subgraph, a specialized pipeline designed to finalize the draft through three sequential steps:

1. **`merge_content`**: Sorts and concatenates the parallelized markdown sections together to form a single, cohesive Markdown document.
2. **`decide_images`**: Analyzes the newly merged text to determine if visual aids (like technical diagrams or tables) would improve the reader's understanding images, inserts `[[IMAGE_X]]` placeholders directly into the markdown draft, and writes highly detailed prompt instructions (`ImageSpec` objects) for each visual.
3. **`generate_and_place_images`**: Interfaces with the HuggingFace Inference API to call the model. It physically generates the images, saves them locally to an `images/` directory, and replaces the temporary `[[IMAGE_X]]` placeholders with standard Markdown image links.

## 🌐 API Layer & Artifact Management

To serve this complex execution graph to the frontend UI, the backend exposes critical features through its FastAPI application:

* **Real-Time Streaming (`/api/generate`)**: Because LangGraph processes (such as executing web research and automated image generation) are highly time-consuming, this endpoint utilizes Server-Sent Events (SSE). This allows the backend to stream live execution updates chunk-by-chunk directly to the user interface, ensuring real-time feedback on the draft's progress.
* **Artifact Retrieval (`/api/blogs`)**: Exposes endpoints to easily list all previously generated blog drafts.
* **Dynamic Asset Bundling (`/api/download/zip`)**: To properly deliver the final product, this endpoint dynamically bundles the finished Markdown document together with its associated generated images into a single, downloadable ZIP archive. This ensures the user receives both the editorial-grade text and the corresponding visual assets in one convenient package, preserving the local Markdown image links.
